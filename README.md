# Droid Party

An iOS app for controlling up to four Sphero Star Wars droids (BB-8, BB-9E, R2-D2, R2-Q5) simultaneously as a fleet. Built on the reverse-engineered Bluetooth stack from [SWSphero](../SWSphero) — no official Sphero SDK, no third-party dependencies.

## What it does

Droids that live plugged in on a shelf, ready to be called into a party at any time. On launch, Droid Party scans for and reconnects every paired droid in parallel. From there you can:

- **Drive each droid individually** — a tab per droid slot with joystick, sound / animation buttons, LED controls, head slider, and leg actions.
- **Broadcast to the whole party** — one tap fires a sound / animation / light effect / sequence on every connected droid at once. Family filter switches between "all", "R-series only", or "BB-series only" so you can send leg actions only to the astromechs.
- **Party Dance** — a built-in coordinated sequence that runs on all four droids in parallel.
- **Emergency Stop All** — one button halts every droid's motion, sound, and effects.

## Architecture

The Bluetooth stack — packet encoding/decoding, protocol handshakes, per-peripheral state machines, drive / capability / sensor controllers — is a verbatim copy of SWSphero's `Core/` and `Models/` directories. See `../SWSphero/README.md` for BLE protocol details.

What's new in Droid Party:

```
DroidParty/
├── ViewModels/
│   ├── FleetViewModel.swift          # Owns the 4 DroidPresences; auto-connect-all;
│   │                                   # UserDefaults-persisted knownDroids map;
│   │                                   # broadcast() fan-out primitive.
│   └── DroidControlViewModel.swift   # Per-droid Operate-screen VM (bound to one presence).
├── Views/
│   ├── ContentView.swift             # 5-tab bottom bar (4 droids + "Party")
│   ├── DroidControlView.swift        # Per-droid Operate screen
│   ├── BroadcastControlView.swift    # "All" tab with party sequences, party lights, e-stop
│   ├── FleetSettingsView.swift       # Pair, forget, reconnect, disconnect droids
│   ├── ControlAtoms.swift            # Reusable icon buttons (extracted from SWSphero)
│   ├── JoystickView.swift            # (copied from SWSphero)
│   └── CalibrationView.swift         # (copied from SWSphero, retyped to DroidControlViewModel)
└── DroidPartyApp.swift               # App entrypoint. @StateObject BLEManager + FleetViewModel.
```

`FleetViewModel.presences` is `[DroidType: DroidPresence]`. Each `DroidPresence` owns its own `DriveController`, `CapabilityController`, `SensorController`, and `SequenceRunner`, sharing a single app-level `BLEManager`. `BLEManager` was already keyed by peripheral UUID throughout, so hosting 4 concurrent connections needed no changes there.

## Requirements

- iOS 17+
- Xcode 16+
- A physical iPhone (BLE isn't available in the simulator)
- Up to 4 Sphero Star Wars droids

## Building

The Xcode project is generated by [xcodegen](https://github.com/yonaskolb/XcodeGen). The pinned xcodegen source is vendored in `.tools/xcodegen-src`; a release binary was built during initial setup.

```sh
# Regenerate the .xcodeproj from project.yml
./.tools/xcodegen-src/.build/release/xcodegen generate

# Open in Xcode
open DroidParty.xcodeproj
```

If you cloned fresh and the release binary is missing:

```sh
(cd .tools/xcodegen-src && swift build -c release)
```

## First-run flow

1. Grant Bluetooth permission when prompted.
2. Open **Fleet Settings** (gear icon in any tab's toolbar), enable **Discovery Mode**.
3. Wake each droid. As they advertise they auto-fill any empty slot matching their type and are remembered for next launch.
4. On subsequent launches the app auto-reconnects to every remembered UUID in parallel.
